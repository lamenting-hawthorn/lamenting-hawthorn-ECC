## 1. Goal & Non-Goals

### Goal
Build a one-way bridge that uses the owner's curated Obsidian vault as the canonical "what matters" filter, pulls supporting evidence from the Hermes SQLite chat history, and writes a small, typed, curated memory graph into Postgres. The result is a `typed_memory` table that mirrors the vault structure plus the SQLite evidence that supports it, with no bloat.

### Non-Goals
- **Not** a full SQLite dump. 28,861 messages are mostly noise (tool output, assistant filler). Only messages that match a vault anchor are imported.
- **Not** an embedding pipeline. `typed_memory.embedding` is left NULL in v1; backfilled later by the existing pipeline.
- **Not** a real-time sync. The bridge is a batch job run on demand (`full` or `incremental`).
- **Not** a vault editor. The bridge is read-only on both sources.
- **Not** an event worker replacement. Vault facts are written directly to `typed_memory` (high confidence, cold data). SQLite evidence may optionally be mirrored to `event_store.events` for audit but the primary write is direct.

## 4. Vault-as-Anchor Model

### 4.1 Core Principle

The vault is the owner's curated "second brain." It already has a structure that maps to the target three-layer model. The bridge does **not** ask "what in SQLite is important?" Instead it asks:

> "For each topic in the vault, what evidence in SQLite supports it?"

This inverts the typical ETL flow. Instead of scanning all 28,861 messages and trying to classify them, we:

1. **Scan the vault** to produce a list of `VaultAnchor` objects.
2. **For each anchor**, extract searchable terms (keywords, dates, project names).
3. **Query SQLite** for messages matching those terms.
4. **Write** a small set of `typed_memory` rows: one for the vault fact itself, plus zero or more for the matched SQLite evidence.
5. **Drop** everything else. If a SQLite message matches no vault anchor, it is not imported.

### 4.2 Why This Works

- The vault has 393 files. Even if every file pulls 20 messages, that is only ~7,860 messages — a 3.7× reduction from 28,861.
- Many vault files are concepts/learning notes that will match few or no SQLite messages (the owner may not have chatted about every ML paper). That is fine — the vault fact is still imported as semantic memory.
- The vault’s folder structure + frontmatter `type` field gives us a near-perfect routing signal for the three-layer model.
- Daily journals provide natural date anchors for episodic memory.

### 4.3 Anchor Strength

Not every markdown file is a strong anchor. The bridge assigns an `anchor_strength` score (0.0–1.0) based on:

| Signal | Strength |
|--------|----------|
| `memory/core.md` | 1.0 (always-load) |
| `memory/factual/*.md` | 0.9 |
| `memory/procedural/*.md` | 0.9 |
| `wiki/personal/*.md` | 0.85 |
| `wiki/projects/*.md` | 0.8 |
| `wiki/entities/*.md` | 0.75 |
| `wiki/learning/*.md` | 0.6 |
| `wiki/concepts/*.md` | 0.5 |
| Daily journal `2026-MM-DD.md` | 0.7 |
| `wiki/company/*.md` | 0.6 |
| `sources/**/*.md` | 0.3 (path pointers, low anchor value) |
| `registry/**/*.md` | 0.4 |
| `_archive/**/*.md` | 0.0 (skipped) |
| `inbox/**/*.md` | 0.2 (unprocessed, skipped by default) |

**Threshold:** `anchor_strength >= 0.5` to pull SQLite evidence. Files below 0.5 are still imported as semantic memory (the vault fact is valid) but no SQLite scan is performed.

### 5.2 Frontmatter Parsing

Use `python-frontmatter` to read YAML frontmatter. Expected fields:

- `title` → used as `summary` in `typed_memory`
- `type` → overrides path-based routing (see §5.3)
- `tags` → added to keyword extraction
- `lastUpdated` / `updated` / `created` → used for date-range matching in SQLite

If frontmatter is malformed (broken YAML, weird unicode), log a warning, treat as no frontmatter, and continue.

### 5.3 Path + Frontmatter Routing Heuristic

The `VaultScanner` produces a `suggested_memory_type` and `suggested_category` for each file:

| File Path Pattern | Frontmatter `type` | `memory_type` | `category` |
|-------------------|-------------------|---------------|------------|
| `memory/core.md` | `core` | `semantic` | `fact` |
| `memory/factual/*.md` | `factual-memory` | `semantic` | `fact` |
| `memory/procedural/*.md` | — | `procedural` | `procedure` |
| `memory/episodic/*.md` | — | `episodic` | `interaction` |
| `wiki/personal/*.md` | `entity` | `semantic` | `preference` |
| `wiki/projects/*.md` | — | `semantic` | `fact` |
| `wiki/entities/*.md` | `entity` | `semantic` | `fact` |
| `wiki/learning/*.md` | — | `semantic` | `fact` |
| `wiki/concepts/*.md` | — | `semantic` | `fact` |
| `wiki/company/*.md` | — | `semantic` | `fact` |
| `2026-*.md` (daily) | — | `episodic` | `interaction` |

**Frontmatter override rule:** If `type` is present and recognized (`core`, `factual-memory`, `entity`, `procedure`, `interaction`), it overrides the path-based guess. Unrecognized `type` values are logged and ignored.

### 5.4 Keyword Extraction

For each anchor, extract a `keyword_set` (list of strings) used to match SQLite messages.

**Extraction strategy:**
1. **File name tokens:** `frubon.md` → `["frubon"]`; `owner-profile.md` → `["owner", "profile"]`
2. **Frontmatter title:** `title: "Project Facts"` → `["project facts"]`
3. **Frontmatter tags:** `tags: [person, identity]` → `["person", "identity"]`
4. **Wikilinks:** `[[frubon|Frubon]]` → `["frubon"]`; `[[growthx|GrowthX]]` → `["growthx"]`
5. **Headers (H1, H2):** `# Frubon` → `["frubon"]`; `## Key Subdirectories` → `["key subdirectories"]`
6. **Bolded terms:** `**Frubon**` → `["frubon"]`
7. **Project paths from core.md:** `<HOME>/Frubon/` → `["frubon"]`

**Stopword filtering:** Remove common English stopwords and Obsidian-specific noise (`MOC`, `README`, `index`).

**Deduplication:** Lowercase, deduplicate, keep original order.

**Example:** `wiki/projects/frubon.md`
```yaml
keyword_set: ["frubon", "fribourne", "qcom", "milk quality", "milk", "quality",
              "optimizer", "frubon-platform", "solids-not-fat", "snf"]
```

### 5.5 Date Range Extraction

For daily journals (`2026-MM-DD.md`), the date is obvious from the filename. For other files, use frontmatter `lastUpdated`, `updated`, or `created`. This date becomes the center of a SQLite search window:

- **Daily journal:** ±1 day (messages from that day and the day after)
- **Project/concept pages:** ±7 days (messages from the week around the last update)
- **Core/personal pages:** ±14 days (broader window, these are always relevant)

If no date is available, no date filter is applied (keyword-only matching).

## 6. SQLite Match Function

### 6.1 Match Dimensions

For each `VaultAnchor`, the `SqliteScanner` queries `messages` using up to three dimensions:

1. **Keyword match** (required): `messages_fts` or `LIKE '%keyword%'`
2. **Date range** (optional): `timestamp BETWEEN start AND end`
3. **Session scope** (optional): if the anchor mentions a known session title or ID

### 6.2 Keyword Match Query

Use the SQLite FTS5 trigram index for robust substring matching:

```sql
SELECT m.id, m.session_id, m.role, m.content, m.timestamp, m.tool_name
FROM messages_fts_trigram fts
JOIN messages m ON fts.rowid = m.id
WHERE fts.content MATCH ?
  AND m.role IN ('user', 'assistant')
  AND m.active = 1
  AND m.compacted = 0
ORDER BY m.timestamp DESC
LIMIT 20;
```

The `?` is bound to a single keyword from the `keyword_set`. The bridge runs this query **once per keyword** in the set, then deduplicates results by `(session_id, message_id)`.

**Why trigram FTS?** It handles substring matches (e.g. "frubon" matches "Frubon" and "frubon-platform"). Standard FTS5 tokenization would miss camelCase.

**Keyword ordering:** Query keywords in order of specificity (longest first, most unique first). Stop after the cumulative match count reaches the per-anchor cap (see §9).

### 6.3 Date Range Filter

After keyword matches are collected, filter by the anchor’s date window:

```python
if anchor.date_center and anchor.date_window_days:
    start = anchor.date_center - timedelta(days=anchor.date_window_days)
    end   = anchor.date_center + timedelta(days=anchor.date_window_days)
    matches = [m for m in matches if start <= m.timestamp <= end]
```

**Rationale:** A keyword like "milk" might match messages about cereal or coffee from unrelated dates. The date window keeps matches relevant to when the vault page was active.

### 6.4 Session Scope Filter

If the anchor’s content contains a wikilink like `[[session:20260610_160214_fb5159]]` (not currently used but supported), restrict matches to that `session_id`.

### 6.5 What Counts as a "Match"?

A message is a match if:
- It contains at least one keyword from the anchor’s `keyword_set` (case-insensitive)
- It falls within the date window (if specified)
- Its `role` is `user` or `assistant` (tool messages are excluded from direct matching; see §7.4 for tool handling)
- Its `content` is not NULL and length > 20 chars
- It is `active = 1` and `compacted = 0`

**Scoring:** Each match gets a `match_score`:
- +1.0 per keyword hit (max 3.0)
- +0.5 if the message is from a `cli` or `subagent` session (higher relevance than `cron`)
- +0.3 if the session `title` contains a keyword
- −0.2 if the message is `assistant` and contains trivial patterns ("ok", "sure", "here you go")

Matches are sorted by `match_score` descending. Only the top N are kept (see §9).

## 7. Three-Layer Routing Rules

### 7.1 Vault Content Routing

| Vault File | `memory_type` | `category` | Rationale |
|------------|---------------|------------|-----------|
| `memory/core.md` | `semantic` | `fact` | Identity, projects, config are stable facts |
| `memory/factual/*.md` | `semantic` | `fact` | Explicitly factual |
| `memory/procedural/*.md` | `procedural` | `procedure` | Explicitly procedural |
| `memory/episodic/*.md` | `episodic` | `interaction` | Explicitly episodic |
| `wiki/personal/*.md` | `semantic` | `preference` | Personal context, communication style |
| `wiki/projects/*.md` | `semantic` | `fact` | Project descriptions, tech stacks |
| `wiki/entities/*.md` | `semantic` | `fact` | Entity definitions |
| `wiki/learning/*.md` | `semantic` | `fact` | "What I learned" |
| `wiki/concepts/*.md` | `semantic` | `fact` | Research concepts, paper summaries |
| `wiki/company/*.md` | `semantic` | `fact` | Company decisions, products, runbooks |
| `2026-*.md` (daily) | `episodic` | `interaction` | Dated journal entries |

**Override by frontmatter `type`:**
- `type: procedure` → `procedural` / `procedure`
- `type: interaction` → `episodic` / `interaction`
- `type: preference` → `semantic` / `preference`
- `type: core` or `type: factual-memory` or `type: entity` → `semantic` / `fact`

### 7.2 SQLite Evidence Routing

For each matched SQLite message, decide `memory_type` and `category` based on message role + content patterns:

| Message Role | Content Pattern | `memory_type` | `category` | Confidence |
|--------------|-----------------|---------------|------------|------------|
| `user` | Contains "remember", "I prefer", "I like", "don’t like", "always use", "never use" | `semantic` | `preference` | 0.7 |
| `user` | Contains "is called", "runs on", "uses", "hosted on", "version", "path is" | `semantic` | `fact` | 0.7 |
| `user` | Contains "no, I meant", "that’s wrong", "actually", "correct that" | `semantic` | `correction` | 0.7 |
| `user` | Explicit task or decision: "do X", "set up Y", "change Z to W" | `episodic` | `action_item` | 0.7 |
| `user` | General chat, question, or discussion | `episodic` | `interaction` | 0.7 |
| `assistant` | Explains a concept, gives factual answer | `semantic` | `fact` | 0.5 |
| `assistant` | Acknowledges, confirms, or gives trivial response | **Dropped** (salience gate) | — | — |
| `tool` | Confirms a system state, file listing, output | `semantic` | `fact` | 0.6 |
| `tool` | Large output (>10KB) with no salient fact | **Dropped** or summarized | — | — |
| `session_meta` | **Always dropped** | — | — | — |

**Session-level aggregation:** If a session has multiple matched messages for the same anchor, collapse them:
- If the session has a clear decision or action item → one `episodic` / `action_item` row + one `semantic` / `fact` row for the decision.
- If the session is a long back-and-forth with no conclusion → one `episodic` / `interaction` row with a summary.
- If the session has a user correction → `semantic` / `correction` row.

### 7.3 Content Patterns (Regex Heuristics)

```python
PREFERENCE_TRIGGERS = {
    "i prefer", "i like", "i don't like", "my favorite",
    "please use", "always use", "never use", "i want", "remember that"
}
FACT_TRIGGERS = {
    "is called", "runs on", "uses", "hosted on", "costs", "version",
    "located at", "url is", "email is", "stored in", "path is", "repo is"
}
CORRECTION_TRIGGERS = {
    "no, i meant", "that's wrong", "actually", "correct that",
    "i meant", "not that", "instead of", "i said"
}
ACTION_TRIGGERS = {
    "set up", "create", "update", "change", "fix", "deploy",
    "run", "execute", "build", "configure", "install"
}
```

These are case-insensitive substring checks (no regex needed for v1).

## 8. PII Redaction

### 8.1 Redaction Strategy

All `content` written to `typed_memory` passes through `src.redaction.pseudonymize_payload()` before insert.

**Redacted entities:**
- Email addresses → `EMAIL_1`, `EMAIL_2`, ...
- IP addresses → `IP_1`, `IP_2`, ...
- Phone numbers → `PHONE_1`, `PHONE_2`, ...
- Account IDs → `ENTITY_A`, `ENTITY_B`, ...
- Long hex IDs → `ID_1`, `ID_2`, ...
- Person names → `NAME_1`, `NAME_2`, ...

**Not redacted:**
- File paths (system facts)
- Tool names (procedural metadata)
- Code snippets (unless they contain PII)
- URLs (unless they contain auth tokens)
- Project names (these are anchor keywords)

### 8.2 Reverse Map Storage

The `PseudonymizedPayload.reverse_mapping` dict is stored in `memory.audit_log`:

```sql
INSERT INTO memory.audit_log (event_type, user_id, session_id, target_id, details)
VALUES (
    'memory_written',
    'owner:<USER>',
    '{session_id_or_vault_path}',
    '{typed_memory_id}',
    jsonb_build_object(
        'reverse_mapping', '{"EMAIL_1": "user@example.com"}'::jsonb,
        'bridge_run_id', 'bridge_20260624_120000',
        'source', 'vault_or_sqlite',
        'redacted_by', 'bridge_vault_and_sessions.py'
    )
);
```

**Recovery query:**
```sql
SELECT details->>'reverse_mapping'
FROM memory.audit_log
WHERE target_id = '{typed_memory_id}'
  AND event_type = 'memory_written';
```

### 8.3 Redaction Scope by Source

| Source | Redaction Applied |
|--------|-------------------|
| Vault files | Yes — even though the owner wrote them, they contain emails/phones |
| SQLite user messages | Yes |
| SQLite assistant messages | Yes |
| SQLite tool messages | Yes (rare but possible) |

## 9. Bloat Prevention

Even with vault filtering, a busy topic could pull hundreds of messages. The design commits to:

### 9.1 Per-Anchor Cap

**Max 20 matched messages per vault page.** After deduplication by `(session_id, message_id)`, if more than 20 remain, keep only the top 20 by `match_score`.

### 9.2 Per-Session Aggregation

If a single session contributes >5 matched messages to the same anchor, collapse them into one `episodic` / `interaction` row with a summary:

```python
if session_matches > 5:
    summary = f"Session '{session_title}' ({session_source}, {message_count} messages). " \
              f"Topics: {', '.join(top_keywords)}. " \
              f"Key action: {extract_action_item(matches)}"
    yield EpisodicMemory(content=summary, ...)
```

### 9.3 Embedding-Based Deduplication (Future / v1.5)

Skip for v1. In v1.5, before inserting a new `typed_memory` row, check if a similar row already exists:

```sql
SELECT id, content
FROM memory.typed_memory
WHERE memory_type = ? AND category = ? AND user_id = 'owner:<USER>'
  AND created_at > now() - interval '30 days'
  AND embedding IS NOT NULL
ORDER BY embedding <-> ?
LIMIT 1;
```

If cosine similarity > 0.9, skip the insert and log a dedup hit.

*(v1 skips this because embeddings are NULL.)*

### 9.4 Tool Message Dedup

Within a session, hash `tool_name + first 200 chars of content`. If the same hash appears >3 times, keep only the first and last occurrence.

## 10. Idempotency Strategy

### 10.1 Vault Imports

**Idempotency key:** `SHA-256(vault_file_path + "|" + content_hash)`

The content hash is computed on the **raw** file contents (before redaction). On re-run:
- If the file has not changed, the idempotency key matches. The bridge skips re-import.
- If the file has changed, the key is new. The bridge inserts the updated fact.
- Old versions are **not** automatically superseded (no versioning in v1). If needed, the owner can manually delete old rows.

**Storage:** The idempotency key is stored in `memory.typed_memory.metadata->>'vault_idempotency_key'`.

### 10.2 SQLite Imports

**Idempotency key:** `SHA-256(source_path + role + content + timestamp)`

Where:
- `source_path` = `<HOME>/.hermes/state.db`
- `role` = message role (`user`, `assistant`, `tool`)
- `content` = raw message content (before redaction)
- `timestamp` = message timestamp as ISO string

**Storage:** For SQLite-derived rows that are also mirrored to `event_store.events`, the key goes into `event_store.events.idempotency_key`. For direct `typed_memory` writes, it goes into `metadata->>'sqlite_idempotency_key'`.

### 10.3 Checkpoint Table

The `memory.bridge_ingest_state` table records the last imported key per source:

| source | last_key | run_id | run_started_at | run_ended_at | stats |
|--------|----------|--------|----------------|--------------|-------|
| `vault` | `memory/core.md\|a3f2...` | `bridge_20260624_120000` | 2026-06-24 12:00:00 | 2026-06-24 12:05:00 | `{"files_scanned": 393, ...}` |
| `sqlite` | `28861` | `bridge_20260624_120000` | 2026-06-24 12:00:00 | 2026-06-24 12:05:00 | `{"messages_matched": 420, ...}` |

For SQLite, `last_key` is the max `messages.id` processed. This allows efficient incremental runs.

### 10.4 Full vs Incremental Mode

- **Full mode:** Re-scan all vault files, re-query all SQLite messages. Uses `ON CONFLICT DO NOTHING` (or equivalent `WHERE NOT EXISTS`) to skip duplicates. Slower but safe.
- **Incremental mode:** Only vault files with mtime > last run, and SQLite messages with `id > last_key`. Fast for daily runs.

### 10.5 Re-run Verification

Running the bridge twice in `full` mode should produce **zero new rows** in `typed_memory` and `audit_log` (except the bridge run tracking entries themselves).

## 11. Failure Handling

### 11.1 Malformed Vault File

**Strategy:** Skip + log, continue.

```python
try:
    post = frontmatter.load(path)
except Exception as e:
    log_error(path, "malformed_frontmatter", str(e))
    stats["vault_errors"] += 1
    continue
```

If the file has broken frontmatter, treat it as plain markdown (no frontmatter overrides). If the file is not valid UTF-8, replace invalid bytes with `�` and log a warning.

### 11.2 Missing SQLite Parent Session

If `sessions.parent_session_id` references a non-existent session, treat it as a fresh session (no continuation). Do not crash.

### 11.3 Postgres Connection Drop Mid-Import

**Strategy:** Transaction per batch of 50 rows.

```python
with psycopg.connect(...) as conn:
    with conn.transaction():
        for item in batch:
            conn.execute(insert_typed_memory, ...)
            conn.execute(insert_audit_log, ...)
        conn.execute(update_checkpoint, ...)
```

If Postgres goes down, the current batch rolls back. The checkpoint is not updated. On restart, the bridge re-processes from the last committed checkpoint. Idempotency keys prevent duplicates.

### 11.4 Embedding Generation Failure

v1 does not generate embeddings. If a future version adds embedding generation and the model is not loaded, store the row with `embedding = NULL` and log a warning. Do not fail the import.

### 11.5 Vault Topic Matches 0 SQLite Messages

This is normal and not a failure. The vault fact is still written to `typed_memory` as a `semantic` / `fact` row. SQLite evidence is optional.

### 11.6 SQLite Query Timeout

If an FTS query takes >5 seconds (unlikely for 28K rows, but possible with very short keywords like "a"), cancel the query, log a warning, and skip the keyword. Move to the next keyword in the set.

## 12. Interaction with the Event Worker

### 12.1 Chosen Strategy: Direct Write (Option A)

The bridge writes directly to `memory.typed_memory` and `memory.audit_log`. It does **not** write to `event_store.events` in v1.

**Justification:**
- The event worker (`event_worker.py`) is designed for runtime chat ingestion (live user → agent → memory). It has known issues (poison-pill deadlock, datetime type mismatch per `STATE.md`).
- The bridge is a cold-data import job. It runs once, produces high-confidence curated memories, and exits. There is no need for async worker processing.
- For personal use, simplicity and reliability trump architectural purity. Direct write is faster and avoids worker bugs.
- If the worker is fixed later, a separate script can re-derive `typed_memory` from SQLite by running the bridge in `--mode=full` again (idempotent).

### 12.2 Future: Hybrid Mode (Option C)

If the owner wants the event store to be the source of truth, a v2 bridge could:
1. Write SQLite evidence to `event_store.events` with `source = 'hermes_bridge'`.
2. Let the worker process it into `typed_memory`.
3. Write vault facts directly (they do not need salience gating).

This is documented here but **not implemented in v1**.

5. **Embeddings in v1:** Should the bridge skip embeddings entirely (NULL), or generate them using the local sentence-transformers model during the bridge run? Skipping is faster; generating makes the memory immediately searchable by vector.
